# Google 商家檔案 SEO Analyzer

## 概述

此技能讓 Claude 能根據使用者提供的 Google Maps 連結，透過 Google Places API (New) 取得真實數據，全面分析該商家的 Google 商家檔案 SEO 狀態，並產出一份精美的 HTML 視覺化報告。

**核心原則：數據驅動，不靠猜測。** 所有分析必須基於 API 回傳的真實數據，不可「推估」或「猜測」。

---

## 工作流程

### Step 1：透過 mcp-google-map MCP Server 取得商家資料

**重要：API 呼叫很珍貴，必須精準使用。每份報告最多 18 次 MCP 呼叫（8 次基礎 + 最多 10 次關鍵字搜尋）。**

#### mcp-google-map MCP Server

透過 MCP 工具直接呼叫 Google Places API (New)，不需自行管理 API Key 和 HTTP 請求。

MCP Server 設定於 `~/.claude/settings.json`（`http://localhost:3000/mcp`），API Key 透過 header 自動帶入。

**可用工具（前綴 `mcp__mcp-google-map__`）：**

| 工具 | 說明 |
|------|------|
| `search_nearby` | 以座標搜尋附近商家，支援關鍵字、距離、評分、營業時間篩選 |
| `get_place_details` | 取得單一商家完整資料（聯絡方式、評論、評分、營業時間） |
| `maps_geocode` | 將地址或商家名稱轉換為座標（經緯度） |
| `maps_reverse_geocode` | 將座標轉換為地址 |
| `maps_distance_matrix` | 計算多點之間的距離與行車時間 |
| `maps_directions` | 取得兩點之間的導航路線 |
| `maps_elevation` | 取得地點的海拔高度 |

#### 呼叫流程（每份報告 4 次）

**呼叫 1：`maps_geocode` — 定位目標商家**

用商家名稱 + 城市取得座標和 Place ID。

```
mcp__mcp-google-map__maps_geocode
→ 輸入：「商家名稱 城市」
→ 取得：座標（lat/lng）+ Place ID
```

從回傳結果比對名稱和地址，選出正確的目標商家。

**呼叫 2：`get_place_details` — 目標商家完整資料**

用 Place ID 取得所有商家欄位。

```
mcp__mcp-google-map__get_place_details
→ 輸入：Place ID
→ 取得：名稱、地址、電話、網站、營業時間、評分、評論、照片、類別、屬性等
```

**呼叫 3：`search_nearby` — 方圓 5km 同類競品**

以目標座標為圓心，半徑 5km，**僅搜尋同類別商家**。

```
mcp__mcp-google-map__search_nearby
→ 輸入：目標座標、distance 5000m、keywords 為商家類別（如 restaurant）
→ 取得：附近同類商家列表（名稱、評分、評論數、距離）
```

若同類別結果太少（< 5 家），改用更通用的類別關鍵字。

**呼叫 4：`search_nearby` — 核心關鍵字競爭排名**

用商家最核心的搜尋關鍵字，查看搜尋排名。

```
mcp__mcp-google-map__search_nearby
→ 輸入：目標座標、distance 5000m、keywords 為「地區 + 服務類型」（如「中壢鹹酥雞」）
→ 取得：前 10-20 名搜尋結果，標記目標商家排名位置
```

最多搜 10 組關鍵字（如「中壢鹹酥雞」「中壢宵夜」「中壢炸雞」等）。

#### 呼叫限制規則

- **每份報告最多 18 次 MCP 呼叫**（8 次基礎 + 最多 10 次關鍵字）
- **不要重複呼叫**同一工具取相同資料
- **`get_place_details` 只呼叫一次** — 一次拿齊所有需要的欄位
- 若某次呼叫失敗，最多重試 1 次
- **所有 MCP 呼叫使用繁體中文**（`languageCode: "zh-TW"`）

#### 補充資料來源（免費，不耗 API）

在 API 取得核心資料後，用 `web_search` 補充：

```
web_search("商家名稱 城市")                → 尋找官網、社群媒體
web_search("商家名稱 評論 推薦")            → 第三方評論（iFoodie、OpenRice 等）
web_search("商家名稱 instagram facebook")  → 社群經營狀況
```

這些搜尋用來：

- 確認 NAP 在不同平台的一致性（特別注意搬遷過的商家）
- 了解線上存在感（部落客食記、社群媒體）
- 發現未連結到 Google 商家檔案的資源

#### 資料可信度分層標示

報告中每個 finding 必須標注資料來源：

| 標籤         | 含義                      | 使用時機                                   |
| ------------ | ------------------------- | ------------------------------------------ |
| `已驗證`   | Places API 回傳的真實數據 | 評分、評論數、類別、電話、網站、營業時間等 |
| `交叉比對` | 多平台比對結果            | NAP 一致性、社群連結                       |

**原則：報告只呈現可驗證的項目。** 無法從外部驗證的項目（Google 貼文、後台成效數據、產品目錄、Logo/封面照）直接跳過，不列入報告。

### Step 2：執行五大面向分析

#### 面向一：商家基本資訊完整度（權重 25%）

**公開評分矩陣（必須在報告中呈現）：**

| 檢查項              | 通過（滿分） | 部分（半分） | 未通過（0 分） | 資料來源                        |
| ------------------- | ------------ | ------------ | -------------- | ------------------------------- |
| 商家名稱與招牌一致  | 10           | 5            | 0              | API `displayName`             |
| 地址完整且格式正確  | 10           | 5            | 0              | API `formattedAddress`        |
| 電話號碼已設定      | 10           | 5            | 0              | API `nationalPhoneNumber`     |
| 營業時間完整        | 10           | 5            | 0              | API `regularOpeningHours`     |
| 主類別精確          | 15           | 8            | 0              | API `primaryType` + `types` |
| 次類別已設定（≥2） | 10           | 5            | 0              | API `types` 陣列長度          |
| 網站連結已設定      | 10           | 5            | 0              | API `websiteUri`              |
| NAP 跨平台一致      | 10           | 5            | 0              | web_search 交叉比對             |

**NAP 一致性子檢查**：用 web_search 搜尋商家名稱，比對 Google 商家檔案與其他平台（iFoodie、OpenRice、Fonfood、Facebook 等）的名稱、地址、電話是否一致。不一致時在報告中列出差異。

#### 面向二：評論分析（權重 30%）

基於 API 回傳的 `rating`、`userRatingCount`、`reviews`（5 則）分析：

- **評論數量等級**：極少 <20 / 偏少 20-50 / 中等 50-150 / 豐富 150-500 / 非常豐富 >500
- **平均評分**：4.5+ 優秀 / 4.0-4.4 良好 / 3.5-3.9 需改善 / <3.5 警示
- **評論關鍵字分析**：從 5 則評論提取高頻正面詞與負面詞
- **商家回覆**：API 不回傳商家回覆，此項不列入報告

提供評論回覆範本（正面 + 負面），自然融入在地關鍵字。

#### 面向三：照片與視覺內容（權重 20%）

基於 API 回傳的 `photos` 陣列（最多 10 張 reference）：

- **照片數量**：API 回傳幾張就報告幾張（註明「API 最多回傳 10 張」）
- **editorialSummary** 有無：反映 Google 是否有足夠內容理解該商家
- Logo / 封面照：API 不區分類型，不列入報告

#### 面向四：商家描述與關鍵字策略（權重 25%）

- **商家描述**：`editorialSummary` 為 null → 確認無描述；有值 → 分析內容
- **在地關鍵字生成**：從商家類型 + 地理位置自動生成 10-15 個關鍵字
- **Google 貼文**：無法從外部驗證，不列入報告
- **線上存在感**：從 web_search 結果評估

#### 面向五（新增）：競爭態勢（不計入總分，獨立呈現）

基於 Nearby Search（5km）和 Text Search（關鍵字排名）的結果：

**5km 競品比較表（必須在報告中呈現）：**

| #   | 商家               | 評分          | 評論數        | 距離         | 有網站       | 照片數      |
| --- | ------------------ | ------------- | ------------- | ------------ | ------------ | ----------- |
| 1   | 競品 A             | 4.6           | 389           | 0.8km        | ✅           | 10          |
| ... | ...                | ...           | ...           | ...          | ...          | ...         |
| ★  | **目標商家** | **4.3** | **212** | **--** | **❌** | **3** |

**關鍵字搜尋排名：**

- 搜尋「地區 + 服務類型」前 10 名，標記目標商家是否在內
- 若不在前 10 → 在報告中強調「搜尋你的核心關鍵字，前 10 名找不到你」

### Step 3：計算總體評分

每個面向根據檢核結果給出 0-100 分，然後按權重計算總分：

```
總分 = 基本資訊(25%) + 評論(30%) + 照片(20%) + 描述與關鍵字(25%)
```

**評分矩陣必須在報告附錄中公開呈現**，讓店家知道分數怎麼來的。

評級制度：

- 🟢 90-100：優秀
- 🟡 70-89：良好
- 🟠 50-69：待改善
- 🔴 0-49：急需改善

### Step 4：產出行動計畫（使用影響力 × 難度矩陣）

**不再使用「高/中/低優先」標籤。** 改用影響力 × 難度矩陣：

|                    | 簡單（< 30min） | 需要時間（1hr+） |
| ------------------ | --------------- | ---------------- |
| **高影響力** | 🔴 立刻做       | 🟡 本週做        |
| **中影響力** | 🟡 本週做       | 🔵 有空再做      |
| **低影響力** | 🔵 有空再做     | ⚪ 可以不做      |

**規則：🔴 立刻做最多只能有 1 件事。** 讓店家知道「今天就做這個」。

每個行動項目包含：

- 做什麼 + 為什麼重要
- 怎麼做（具體步驟）
- 難度星級（⭐ ~ ⭐⭐⭐）
- 預估時間
- 雙層式指引：🟢 自己做 / 🔴 交給專家

**每個建議只在報告中出現一次。** 不在摘要、面向分析、行動計畫中重複。

### Step 5：產出 HTML 報告

**每次分析都必須遵循統一的 HTML 模板**，確保報告風格一致、專業。

#### 模板使用方式

1. 先用 `view` 工具讀取 `references/template.html`（位於本技能目錄下）
2. 以模板為基礎，將 `{{placeholder}}` 替換為實際分析數據
3. **嚴格保留模板中的所有 CSS 變數、class 命名、HTML 結構**，不可自行修改設計風格
4. 僅在資料數量上做增減（如更多 finding-card），不改變佈局邏輯

#### 報告結構（每個資訊只出現一次）

| 順序 | 區塊             | 對應 class                | 說明                                              |
| ---- | ---------------- | ------------------------- | ------------------------------------------------- |
| 1    | Hero 標頭        | `.hero`                 | 商家名稱、地址、電話、分析日期 +`.hero-creator` |
| 2    | 錨點導航列       | `.sticky-nav`           | 快速跳轉所有區塊                                  |
| 3    | 評分儀表板       | `.dashboard`            | 總分圓環 + 四面向分數卡                           |
| 4    | 30 秒摘要        | `.exec-summary`         | 白話總結 + 最該先做的 1 件事 + 好消息             |
| 5    | 競爭態勢（新增） | `.section#sec-compete`  | 5km 競品表 + 關鍵字排名                           |
| 6    | 診斷發現         | `.section` ×4          | 四面向分析，每項標注資料來源                      |
| 7    | 行動計畫         | `.section#sec-action`   | 影響力×難度矩陣 + 每個行動只出現一次             |

1. **新增「競爭態勢」區塊**（順序 5）— 5km 競品表 + 關鍵字排名，基於 API 真實數據
2. **CTA 只放在最後**（`.about-section`）— 不在報告中間插入 CTA
3. **行動計畫獨立成一個區塊** — 每個建議只出現一次，不在面向分析中重複
4. **附錄整合** — 評分矩陣、copy-paste 範本、名詞解釋都放附錄
5. **面向分析精簡** — 只列有問題的項目，通過項目極簡呈現（一行帶過）

#### 客戶導向 UX 規範

**UX-1 — 30 秒摘要精簡化**

- Dashboard 之後，用 3-4 句白話文說明現況
- 只列「最該先做的 1 件事」（不是 3 件），附預估時間

**UX-2 — 術語白話化 + Tooltip**

- 專業術語用 `.term` + `.tip` tooltip
- 必須解釋：Google 商家檔案、SEO、Google 貼文、NAP 一致性、關鍵字、搜尋曝光次數

**UX-3 — Finding Card 標注資料來源**

- 每張 `.finding-card` 必須標注「已驗證 / 交叉比對」

- **通過項目極簡呈現**（一行），不需要長篇解釋為什麼好

**UX-4 — 行動計畫使用矩陣排列**

- 用影響力 × 難度矩陣取代「高/中/低優先」
- 🔴 立刻做最多 1 件

**UX-5 — 競品表視覺突出**

- 目標商家用 ★ 標記、粗體、底色區分
- 表格按評分或熱門度排序
- 一句話點出差距：「你的照片數在 20 家中排倒數第 3」

**UX-6 — 重新分析 CTA**：報告尾端提供明確 CTA，引導用戶預約諮詢或免費索取新報告，含 30 天見效承諾

## 處理邊界情況

- **Google Maps 短網址無法解析**：請使用者提供商家名稱 + 城市，用 Text Search
- **API 回傳多筆結果**：用名稱 + 地址比對，選出正確目標
- **API 呼叫失敗**：最多重試 1 次，仍失敗則用 web_search 替代
- **多個商家**：每個產出獨立報告，API 呼叫分開計算
- **非繁體中文商家**：報告仍以繁體中文撰寫

---

## 語言與語調

- 繁體中文，以「商家老闆看得懂」為最高原則
- 使用「您」稱呼商家主
- 摘要用比喻具體化（「像一張空白名片」）
- **不囉嗦**：通過的項目一行帶過，問題項目才詳細說明
- **不重複**：每個建議在整份報告中只出現一次
