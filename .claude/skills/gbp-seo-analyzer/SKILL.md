# Google 商家檔案 SEO Analyzer

## 概述

此技能讓 Claude 能根據使用者提供的 Google Maps 連結，全面分析該商家的 Google 商家檔案 SEO 狀態，並產出一份精美的 HTML 視覺化報告，包含：

- 各面向的評分與診斷（含 NAP 一致性檢查）
- 在地關鍵字策略與生成建議
- 具體可執行的改善策略（含評論回覆範本）
- 量化成功指標與觀察時程
- LocalBusiness 結構化資料建議

---

## 工作流程

### Step 1：取得商家資料

收到 Google Maps 連結或商家資訊後，按以下優先順序取得資料：

#### 方法一（優先）：使用 `places_search` 工具

這是最可靠且資料最完整的方式。`places_search` 透過 Google Places API 回傳結構化資料。

```
places_search(queries=[
  {"query": "商家名稱 城市", "max_results": 5}
])
```

從回傳結果中直接擷取：
- `name`：商家名稱
- `address`：完整地址
- `phone_number`：電話號碼（null 代表未設定 → 標記為缺失）
- `rating` / `rating_count`：平均評分與評論數量
- `types`：商家類別陣列（判斷類別是否精準）
- `weekday_hours`：營業時間（空陣列 → 標記為缺失）
- `reviews`：近期評論內容（用於評論關鍵字分析）
- `website_url`：官網連結
- `place_id`：Google Place ID（可供後續追蹤）

**重要**：回傳結果可能包含多筆不相關的商家，務必透過名稱和地址比對，選出正確的目標商家。

#### 方法二（補充）：使用 `web_search` 搜尋更多公開資訊

在 `places_search` 取得基礎資料後，用 `web_search` 補充以下資訊：

```
web_search("商家名稱 城市")                → 尋找官網、社群媒體
web_search("商家名稱 評論 推薦")            → 第三方評論與部落格文章
web_search("商家名稱 instagram facebook")  → 社群經營狀況
```

這些額外搜尋用來：
- 確認 NAP（名稱、地址、電話）在不同平台的一致性
- 了解商家在 Google 以外的線上存在感
- 發現商家可能已有但未連結到 Google 商家檔案的資源

#### 方法三（回退）：處理 Google Maps 短網址

Google Maps 短網址（maps.app.goo.gl）會被 robots.txt 擋住，`web_fetch` 無法直接抓取。遇到此情況：

1. **不要反覆嘗試 `web_fetch`**，這會浪費時間
2. 嘗試用 `web_search` 搜尋短網址中的 ID（如 `"VujrzA57ogVArTgm6" google maps`）
3. 若仍無法解析，**立即請使用者提供商家名稱 + 所在城市**，然後回到方法一

#### 無法遠端偵測的項目

以下項目需根據已有資訊推斷，或在報告中標註「無法遠端偵測，建議手動確認」：

- 商家描述完整內容（後台才能看到）
- 照片確切數量與類別分布
- 商家回覆率與回覆品質
- Google 貼文使用狀況
- 產品/服務/菜單是否已建立
- NAP 在其他平台（Facebook、ibon、愛食記等）的一致性

這些項目仍應列入分析報告中，並在建議中提醒使用者自行確認。

### Step 2：執行四大面向分析

若某些資料無法取得，應標註為「無法偵測，建議手動確認」，而非跳過。

#### 面向一：商家基本資訊完整度（權重 25%）

**檢核項目與評分標準：**

| 項目 | 完整（滿分） | 部分完整（半分） | 缺失（0 分） |
|------|-------------|-----------------|-------------|
| 商家名稱（含關鍵字策略） | 名稱精確反映業務且含地區/服務關鍵字 | 名稱正確但缺少關鍵字 | 不明確或過於籠統 |
| 完整地址 | 地址完整、格式正確 | 部分缺失 | 未提供 |
| 電話號碼 | 本地號碼、格式正確 | 有但格式不佳 | 未提供 |
| 營業時間 | 每日時段完整、含特殊時段 | 僅部分天數 | 未設定 |
| 商家類別 | 主類別精準 + 有相關次類別 | 僅主類別 | 類別不精準 |
| 網站連結 | 有且可正常連結 | 有但連結異常 | 未提供 |
| 服務區域/菜單/服務項目 | 已設定且完整 | 部分設定 | 未設定 |

**NAP 一致性子檢查**（若從 web_search 發現其他平台資料）：

對比 Google 商家檔案與其他來源（官網、Facebook、愛食記、ibon 等）的名稱、地址、電話格式是否一致。不一致時在報告中列出差異並建議標準化格式：

```
標準 NAP 格式：
- 名稱：使用正式登記名稱，各平台完全一致
- 地址：[縣市][區][路/街][號]（統一全形/半形、有無「之」）
- 電話：統一格式 04-XXXX-XXXX 或 +886-4-XXXX-XXXX
```

#### 面向二：評論分析（權重 30%）

**分析重點：**
- **評論數量**：極少 <20 / 偏少 20-50 / 中等 50-150 / 豐富 150-500 / 非常豐富 >500
- **平均評分**：4.5+ 優秀 / 4.0-4.4 良好 / 3.5-3.9 需改善 / <3.5 警示
- **近期評論趨勢**：最近的評論是正面還是負面？是否有趨勢變化？
- **商家回覆率與品質**：是否有回覆？回覆是否個人化？是否包含關鍵字？回覆時效？
- **評論關鍵字分析**：從評論提取高頻正面詞與負面詞，識別顧客最在意的面向

**評論回覆範本**（在建議中提供）：

正面評論回覆範本：
```
「[顧客名]您好，非常感謝您的好評！很高興我們的[服務/產品]讓您滿意。
我們會持續努力提供最好的[服務類型]，期待下次為您服務！
—— [商家名稱] 團隊」
```

負面評論回覆範本：
```
「[顧客名]您好，非常抱歉這次的體驗未達您的期待。
我們已經記下您的回饋，並會在[具體改善方向]上加以改進。
歡迎您私訊或致電 [電話] 與我們聯繫，我們希望有機會為您重新服務。
—— [商家名稱] 團隊」
```

**評論獲取策略建議**（在 rec-card 中呈現）：
- 製作評論邀請卡（附 QR code 直連 Google 評論頁面）
- 服務/出貨完成後 24 小時內發送感謝訊息 + 評論連結
- 在收據/包裹中附上「留評送小禮」提醒（注意：Google 禁止直接以金錢換評論）

#### 面向三：照片與視覺內容（權重 20%）

**評估項目：**
- 照片總數量（基準：至少 10 張，理想 25+ 張）
- 是否有 Logo 照片與封面照片
- 照片類別多樣性（外觀、內部、產品/菜單、團隊、氛圍）
- 照片品質觀察（基於可獲得的資訊推斷）
- 是否有近期更新的照片
- 用戶上傳照片 vs 商家上傳照片的比例

**照片上傳建議清單**（在 rec-card 步驟中具體化）：
```
建議照片組合（最少 10 張）：
- 店面外觀 2-3 張（含招牌、入口）
- 內部環境 2-3 張（座位、裝潢、氛圍）
- 產品/菜單 3-5 張（招牌商品特寫）
- 團隊照片 1-2 張（增加信任感）
- 服務過程 1-2 張（顯示專業度）
- 證書/獎牌 1 張（如有）
```

#### 面向四：商家描述與關鍵字策略（權重 25%）

**評估項目：**
- 商家描述是否存在且充分利用字數（上限 750 字元）
- 描述中是否自然融入目標關鍵字（產業 + 地區 + 服務）
- 是否提及獨特賣點（USP）
- 描述的語調是否符合品牌形象
- 是否善用 Google 貼文（Posts）功能
- 產品/服務/菜單是否已在 Google 商家檔案中完整建立

**在地關鍵字生成**（在建議中提供具體關鍵字清單）：

從商家的服務類型和地理位置自動生成 5 種模式的在地關鍵字：

```
模式 1：「服務 + 城市」       → 台中 咖啡廳
模式 2：「城市 + 服務 + 推薦」 → 台中咖啡廳推薦
模式 3：「最佳/好評 + 服務 + 在 + 區域」 → 西屯區好評咖啡廳
模式 4：「服務 + 社區/商圈」   → 逢甲咖啡廳
模式 5：「服務 + 特色」       → 台中手沖咖啡、台中寵物友善咖啡
```

為每個面向建議至少 10 個在地關鍵字，區分高搜尋量（3-5 個）、中搜尋量（3-5 個）、長尾關鍵字（3-5 個）。

**商家描述撰寫範本**（在 rec-card 中提供）：

若商家缺少描述，直接提供一段 500-750 字元的建議描述，自然融入關鍵字：
```
結構：
第 1 段 — 我們是誰 + 核心服務 + 地區（含主要關鍵字）
第 2 段 — 獨特賣點 + 為什麼選擇我們
第 3 段 — 服務項目清單（含次要關鍵字）
第 4 段 — 行動呼籲（來電/來店/線上預約）
```

### Step 3：計算總體評分

每個面向根據檢核結果給出 0-100 分，然後按權重計算總分：

```
總分 = 基本資訊(25%) + 評論(30%) + 照片(20%) + 描述與關鍵字(25%)
```

評級制度：
- 🟢 90-100：優秀（持續維護即可）
- 🟡 70-89：良好（有明確優化空間）
- 🟠 50-69：待改善（需要系統性優化）
- 🔴 0-49：急需改善（存在嚴重缺漏）

---

### Step 4：產出改善策略

根據分析結果，為每個面向提供：

1. **立即行動（第 1-2 週）**：短期可完成的修改，如補齊資訊、更新照片
2. **短期優化（第 3-4 週）**：需要準備的改善，如撰寫描述、建立回覆策略
3. **持續經營（每月）**：長期執行策略，如定期更新貼文、爭取評論

每個行動都要包含：
- 具體做什麼（What）
- 為什麼重要（Why）
- 怎麼做（How）— 具體步驟
- 預期效果與觀察指標

**額外輸出項目**（在建議中自然穿插）：
- 為缺少描述的商家直接撰寫建議描述（500-750 字元）
- 為評論回覆率低的商家提供正面/負面回覆範本
- 為關鍵字不足的商家列出 10-15 個在地關鍵字建議
- 建議商家在官網加入 LocalBusiness Schema 結構化資料（若有官網）

---

### Step 5：定義觀察期與成功指標

根據改善策略設定觀察期：

| 階段 | 時間區間 | 觀察重點 |
|------|----------|----------|
| 基礎優化期 | 第 1-4 週 | 資訊完整度提升、照片增加、描述優化 |
| 成效醞釀期 | 第 5-8 週 | 搜尋曝光次數變化、新評論趨勢 |
| 穩定成長期 | 第 9-12 週 | 排名變化、來電/導航次數趨勢 |
| 長期追蹤期 | 第 13 週起（每月） | 整體流量趨勢、競爭對手比較 |

**量化成功指標**（在報告的行動時間軸區塊中呈現）：

| 時程 | 指標 | 目標 |
|------|------|------|
| 1 個月 | Google 商家檔案完整度 | > 90%（所有欄位填滿） |
| 1 個月 | 評論回覆率 | 100%（所有評論都有回覆） |
| 3 個月 | 搜尋曝光次數 | 較優化前 +50% |
| 3 個月 | 新增評論數 | +10-20 則 |
| 6 個月 | 主要關鍵字排名 | 地區搜尋前 5 名 |
| 6 個月 | 來電/導航次數 | 較優化前 +30% |

### Step 6：產出 HTML 報告

**每次分析都必須遵循統一的 HTML 模板**，確保報告風格一致、專業，且對不懂 SEO 的商家老闆來說容易理解。

#### 模板使用方式

1. 先用 `view` 工具讀取 `references/template.html`（位於本技能目錄下）
2. 以模板為基礎，將 `{{placeholder}}` 替換為實際分析數據
3. **嚴格保留模板中的所有 CSS 變數、class 命名、HTML 結構**，不可自行修改設計風格
4. 僅在資料數量上做增減（如更多 finding-card、更多 timeline-action），不改變佈局邏輯

#### 模板固定結構（不可變更）

報告由以下 9 個區塊按順序組成，每次都必須完整呈現：

| 順序 | 區塊 | 對應 class | 說明 |
|------|------|-----------|------|
| 1 | Hero 標頭 | `.hero` | 商家名稱、地址、電話、分析日期 + `.hero-creator` 品牌標示 |
| 2 | 錨點導航列 | `.sticky-nav` | Sticky 定位，快速跳轉所有區塊 |
| 3 | 評分儀表板 | `.dashboard` | 左側總分圓環 + 右側四面向分數卡（含白話狀態描述） |
| 4 | 30 秒摘要 | `.exec-summary` | 白話總結 + 最該先做的 3 件事（含預估時間） |
| 5 | 面向一 + 面向二分析 | `.section` ×2 | verdict → findings → rec-card |
| 6 | AI 諮詢 CTA | `.ai-cta-section` | 「想了解如何運用 AI 分析你的商家？」→ Potaly 官網 1hr NT$4,000 |
| 7 | 面向三 + 面向四分析 | `.section` ×2 | verdict → findings → rec-card |
| 8 | 行動時間軸 | `.timeline` | 三階段 + 互動 checkbox + 成功指標 |
| 9 | 名詞小辭典 | `.glossary` | 所有術語白話解釋 |
| 10 | 自我介紹 + CTA | `.about-section` | 個人介紹 + 3 張 CTA 卡片（含 Potaly） |
| 11 | 14 天跟進預告 | `.followup-notice` | 「14 天後我們會回來看看你的進步！」 |
| 12 | Footer | `.report-footer` | 分析日期 + 免責聲明 + Created by @dev.from.zero |

#### 品牌標示規範（每份報告必須包含）

**Hero 區塊**：底部加入 `.hero-creator` 連結，文字為「Analysed by @dev.from.zero」，連結至 `https://www.threads.com/@dev.from.zero`

**AI 諮詢 CTA（`.ai-cta-section`）**：位於面向二和面向三之間（報告中段）
- 用戶剛看完兩個面向的分析，正好奇「這是怎麼做到的」，轉換時機最佳
- 標題：「想了解如何運用 AI 分析你的商家？」
- 副標題：「1 小時深度諮詢，教你用 AI 工具持續追蹤與優化」
- CTA 按鈕：「預約 AI 諮詢 — NT$4,000 / 1hr」→ 連結導向 Potaly 官網
- 翡翠綠底，視覺突出但不突兀

**自我介紹 + CTA（`.about-section`）**：位於名詞小辭典之後
- 左側：真人頭像（圓形 `<img>`）+ 名稱「dev.from.zero」+ 簡介
- 簡介固定內容：「懶惰的工程師」開頭，交流主題包含 SaaS 開發、工具、AI 自動化、NoCode、SEO 優化，歡迎交流與專案討論
- 右側：3 張 CTA 卡片
  - 🤖 1hr AI 諮詢 NT$4,000 → Potaly 官網
  - 🌐 Google 商家 SEO 優化服務 → `https://gbp-seo.devfromzero.xyz`
  - 💬 追蹤 Threads @dev.from.zero → `https://www.threads.com/@dev.from.zero`
- 深藍底白字，與 Hero 風格一致

**14 天跟進預告（`.followup-notice`）**：位於 about-section 之後、Footer 之前
- 標題：「14 天後我們會回來看看你的進步！」
- 語氣溫暖，傳達關心而非推銷
- 淺綠底色，與行動時間軸呼應

**Footer**：「Created by @dev.from.zero」文字連結 + 網站按鈕（不重複放 Threads 按鈕）

#### 客戶導向 UX 規範（7 項，全數必須遵循）

**UX-1 — 30 秒摘要卡片（`.exec-summary`）**
- Dashboard 之後、四面向之前，插入翡翠綠底摘要區塊
- 用 3-4 句白話文說明：「目前狀況」「最大問題」「好消息」
- 列出最該先做的 3 件事，每件附預估時間

**UX-2 — 術語白話化 + Tooltip**
- 專業術語首次出現時，使用 `.term` class 包裹，內含 `.tip` 子元素
- Hover/Tap 時顯示解釋泡泡
- 報告末尾必須有「名詞小辭典」（`.glossary`）
- 行文中用白話替代：「Google 商家檔案（就是您在 Google 地圖上看到的店家頁面）」
- 必須解釋的術語：Google 商家檔案、SEO、Google 貼文、服務區域型商家、Insights、關鍵字、產品目錄、搜尋曝光次數、NAP 一致性、結構化資料

**UX-3 — Finding Card 加「對您的影響」**
- 每張 `.finding-card` 底部必須有 `.finding-impact`
- 用 💡 開頭，一句白話說明對商家生意的具體影響
- 通過項目用 ✅ 開頭，強調這是優勢

**UX-4 — 建議步驟視覺化（Step Cards）**

## 處理邊界情況

- **連結無法解析**：請使用者提供商家名稱 + 地區，改用 `places_search`
- **資料不足**：明確標註無法遠端偵測的項目，建議使用者提供後台截圖
- **多個連結**：每個商家產出獨立報告，最後加入比較摘要
- **非繁體中文商家**：報告仍以繁體中文撰寫，商家名稱與引用保留原文
- **已有官網的商家**：額外建議加入 LocalBusiness Schema 結構化資料

## 語言與語調

- 報告全程使用**繁體中文**
- 語調專業但友善，**以「商家老闆看得懂」為最高原則**
- 避免直接使用專業術語，首次出現一律加白話解釋（行內括號 + tooltip）
- 改善建議要具體、可操作，拆成步驟卡片
- 每個問題都要說明「對您的影響」— 用商家的角度解釋為什麼重要
- 使用「您」稱呼商家主
- 每個面向開頭用一句話結論（`.section-verdict`）
- 摘要區用比喻讓抽象概念具體化（例如「像一張空白名片」「像沒有招牌的店」）
