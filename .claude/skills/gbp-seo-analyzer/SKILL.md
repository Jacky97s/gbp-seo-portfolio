# Google 商家檔案 SEO Analyzer

## 概述

此技能讓 Claude 能根據使用者提供的 Google Maps 連結，透過 Google Places API (New) 取得真實數據，全面分析該商家的 Google 商家檔案 SEO 狀態，並產出一份精美的 HTML 視覺化報告。

**核心原則：數據驅動，不靠猜測。** 所有分析必須基於 API 回傳的真實數據，不可「推估」或「猜測」。

> 📖 分析過程中隨時參考 `references/local-seo-knowledge.md`，內含演算法原理、產業別實戰經驗、評論策略、停權風險判斷等 Local SEO 領域知識，用於強化分析深度與建議品質。

---

## 工作流程

### Step 1：透過 Google Places API (New) 取得商家資料

**重要：Google Places API (New) 是本技能取得商家數據的唯一且優先方式。** 不使用任何 MCP Server，直接透過 Bash 工具執行 `curl` 呼叫 REST API。

**每份報告最多 18 次 API 呼叫（8 次基礎 + 最多 10 次關鍵字搜尋）。**

> 📖 API 端點、FieldMask、呼叫流程等技術細節，請用 `Read` 工具載入 `references/api-guide.md`。

---

### Step 2：執行五大面向分析

#### 面向一：商家基本資訊完整度（權重 25%）

> 📖 參考知識庫「二、商家名稱與產業類別」— 名稱規範、停權風險、產業類別對應後台功能。

**公開評分矩陣（必須在報告中呈現）：**

| 檢查項              | 通過（滿分） | 部分（半分） | 未通過（0 分） | 資料來源                        |
| ------------------- | ------------ | ------------ | -------------- | ------------------------------- |
| 商家名稱與招牌一致  | 10           | 5            | 0              | API `displayName`             |
| 地址完整且格式正確  | 10           | 5            | 0              | API `formattedAddress`        |
| 電話號碼已設定      | 10           | 5            | 0              | API `nationalPhoneNumber`     |
| 營業時間完整        | 10           | 5            | 0              | API `regularOpeningHours`     |
| 主類別精確          | 15           | 8            | 0              | API `primaryType` + `types` |
| 次類別已設定（≥2） | 10           | 5            | 0              | API `types` 陣列長度          |
| 網站連結已設定      | 10           | 5            | 0              | API `websiteUri`              |
| NAP 跨平台一致      | 10           | 5            | 0              | web_search 交叉比對             |

**名稱停權風險檢查**：分析 `displayName` 是否包含明顯的關鍵字堆砌（如「小明雞肉飯｜台北最好吃」）。名稱塞關鍵字是目前約八成停權案件的原因，若偵測到需在報告中以 ⚠️ 警告標示。

**產業類別對應後台功能**：根據 `primaryType` 判斷產業類型，在建議中提醒對應的後台功能操作（餐飲→菜單、零售→產品、服務→服務項目、旅宿→16 項飯店詳情、景點→門票）。

**NAP 一致性子檢查**：用 web_search 搜尋商家名稱，比對 Google 商家檔案與其他平台（iFoodie、OpenRice、Fonfood、Facebook 等）的名稱、地址、電話是否一致。不一致時在報告中列出差異。特別注意搬遷過的商家，舊地址可能在各平台殘留。

#### 面向二：評論分析（權重 30%）

> 📖 參考知識庫「三、評論經營」及 `references/api-guide.md`「評論深度收集策略」。

**評論收集：** 使用 `scripts/scrape-reviews.py --place-id "${PLACE_ID}" --sort all --limit 50` 取得大量評論（需 `SERP_API_KEY`，約 16 次 SerpAPI 呼叫，去重後 ~160 則）。若無 key 則退回 Places API 5 則 + web_search。報告標注「基於 N 則評論分析」。

**分析維度**（8 項）：數量等級、星級分布、最高分分析、最低分分析、最新評論趨勢、商家回覆分析、評論關鍵字、在地嚮導比例。

> 📖 各維度的詳細操作指引見 `references/api-guide.md`「評論分析維度」。

**平均評分**：4.5+ 優秀 / 4.0-4.4 良好 / 3.5-3.9 需改善 / <3.5 警示

#### 面向三：照片與視覺內容（權重 20%）

基於 API 回傳的 `photos` 陣列（最多 10 張 reference）：

- **照片數量**：API 回傳幾張就報告幾張（註明「API 最多回傳 10 張」）
- **editorialSummary** 有無：反映 Google 是否有足夠內容理解該商家
- Logo / 封面照：API 不區分類型，不列入報告

#### 面向四：商家描述與關鍵字策略（權重 25%）

> 📖 參考知識庫「一、演算法基礎」的關鍵字分類（流量字 vs 轉換字）及「八、外部資訊與品牌建設」。

- **商家描述**：`editorialSummary` 為 null → 確認無描述；有值 → **在報告中直接顯示 Google 的 Editorial Summary 原文**，並分析其內容是否涵蓋核心關鍵字
- **在地關鍵字生成**：從商家類型 + 地理位置自動生成 10-15 個關鍵字
  - 區分**流量字**（搜尋量高、拼曝光）和**轉換字**（明確需求、轉換率高）
  - 優先建議「**地區 + 服務**」組合（自帶搜尋量，競爭較低）
- **Google 貼文**：無法從外部驗證，不列入報告，但在行動建議中可提及貼文功能權重正在提升的趨勢
- **線上存在感**：從 web_search 結果評估（部落客、論壇、新聞、官網、社群）
- **外部資訊評估**：外部流量來源（部落客文章、論壇提及、新聞報導、官網 SEO）越豐富，排名越好

#### 面向五（新增）：競爭態勢（不計入總分，獨立呈現）

基於 Nearby Search（5km）和 Text Search（關鍵字排名）的結果：

**5km 競品比較表（必須在報告中呈現）：**

| #   | 商家               | 評分          | 評論數        | 距離         | 有網站       | 照片數      |
| --- | ------------------ | ------------- | ------------- | ------------ | ------------ | ----------- |
| 1   | 競品 A             | 4.6           | 389           | 0.8km        | ✅           | 10          |
| ... | ...                | ...           | ...           | ...          | ...          | ...         |
| ★  | **目標商家** | **4.3** | **212** | **--** | **❌** | **3** |

**關鍵字搜尋排名：**

- 搜尋「地區 + 服務類型」前 10 名，標記目標商家是否在內
- 若不在前 10 → 在報告中強調「搜尋你的核心關鍵字，前 10 名找不到你」

---

### Step 3：計算總體評分

每個面向根據檢核結果給出 0-100 分，然後按權重計算總分：

```
總分 = 基本資訊(25%) + 評論(30%) + 照片(20%) + 描述與關鍵字(25%)
```

**評分矩陣必須在報告附錄中公開呈現**，讓店家知道分數怎麼來的。

評級制度：

- 🟢 90-100：優秀
- 🟡 70-89：良好
- 🟠 50-69：待改善
- 🔴 0-49：急需改善

---

### Step 4：產出行動計畫（使用影響力 × 難度矩陣）

> 📖 參考知識庫「五、產業實戰成效基準」根據商家產業類型調整建議優先序；「四、停權與驗證」在建議中加入停權風險提醒；「六、AI 時代的商家經營」加入前瞻性建議。

**不再使用「高/中/低優先」標籤。** 改用影響力 × 難度矩陣：

|                    | 簡單（< 30min） | 需要時間（1hr+） |
| ------------------ | --------------- | ---------------- |
| **高影響力** | 🔴 立刻做       | 🟡 本週做        |
| **中影響力** | 🟡 本週做       | 🔵 有空再做      |
| **低影響力** | 🔵 有空再做     | ⚪ 可以不做      |

**規則：🔴 立刻做最多只能有 1 件事。** 讓店家知道「今天就做這個」。

每個行動項目包含：

- 做什麼 + 為什麼重要
- 怎麼做（具體步驟）
- 難度星級（⭐ ~ ⭐⭐⭐）
- 預估時間
- 雙層式指引：🟢 自己做 / 🔴 交給專家

**產業化建議**：根據商家 `primaryType` 提供對應操作重點（如餐飲→菜單關鍵字、服務業→服務項目完善、旅宿→飯店詳情 16 項），並引用同產業成效案例作為預期效果佐證。

**停權風險提醒**：若發現名稱塞關鍵字、營業時間可能異常等風險，在行動計畫中以 ⚠️ 優先標示，因為停權的損失遠大於排名優化的收益。

**前瞻性建議（選擇性加入）**：AI Mode 優化、YouTube Short 串接商家 + Geotag、Google 貼文關鍵字趨勢等。

**每個建議只在報告中出現一次。** 不在摘要、面向分析、行動計畫中重複。

---

### Step 5：產出 HTML 報告

**每次分析都必須遵循統一的 HTML 模板**，確保報告風格一致、專業。

> 📖 報告結構、品牌標示、UX 規範、設計規格等細節，請用 `Read` 工具載入 `references/report-spec.md`。
> 📖 HTML 模板本體，請用 `Read` 工具載入 `references/template.html`。

將完成的 HTML 檔案儲存到專案根目錄的 `reports/` 資料夾，命名格式為 `{ID}.html`。
- **ID**：由使用者提供的唯一識別碼（如 `MMT5JQXK`），若使用者未提供則詢問
- 範例：`MMT5JQXK.html`

## 地域化適配

**根據商家所在國家/地區自動調整分析策略。** 從 API 回傳的 `formattedAddress` 和 `addressComponents` 判斷國家。

> 📖 各國 NAP 比對平台、社群平台、文化差異等查找表，請用 `Read` 工具載入 `references/localization.md`。

## 處理邊界情況

- **Google Maps 短網址無法解析**：請使用者提供商家名稱 + 城市，用 `/places:searchText` 端點搜尋
- **API 回傳多筆結果**：用名稱 + 地址比對，選出正確目標
- **API 呼叫失敗**：最多重試 1 次，仍失敗則用 web_search 替代（curl 加 `-k` 嘗試繞過 SSL）
- **多個商家**：每個產出獨立報告，API 呼叫分開計算
- **非繁體中文商家**：報告仍以繁體中文撰寫

---

## 語言與語調

- 繁體中文，以「商家老闆看得懂」為最高原則
- 使用「您」稱呼商家主
- 摘要用比喻具體化（「像一張空白名片」）
- **不囉嗦**：通過的項目一行帶過，問題項目才詳細說明
- **不重複**：每個建議在整份報告中只出現一次
